# Writing Guide

## Core Principle
Every output is a Signal: S = (Mode, Genre, Type, Format, Structure).
Before writing, resolve all 5 dimensions. Unresolved dimensions create noise.

## Genre Quick Reference

### brief
**Receiver**: Salespeople, executives, clients, non-technical
**Goal**: Compel action
**Skeleton**:
1. Objective — One sentence. What outcome.
2. Key Messages — 3-5 bullets. Each an atomic claim.
3. Call to Action — Single unambiguous ask + deadline.
4. Supporting Materials — Links if needed.

**Example receivers**: Robert Potter, Len, Liam, Jared, Bennett, Jordan, Tom

### spec
**Receiver**: Developers, technical partners
**Goal**: Define what to build
**Skeleton**:
1. Goal — What and why.
2. Requirements — Numbered list.
3. Constraints — What's off the table.
4. Architecture — How it fits.
5. Acceptance Criteria — How we know it's done.

**Example receivers**: Developers, lead engineer, technical partner

### plan
**Receiver**: The operator (self)
**Goal**: Structured execution
**Skeleton**:
1. Objective — What this plan achieves.
2. Non-Negotiables — Top 3, no exceptions.
3. Time Blocks — Day-by-day allocation.
4. Dependencies — Who/what we're waiting on.
5. Success Criteria — How we measure the week.

### transcript
**Receiver**: Internal processing
**Goal**: Extract signal from conversation
**Skeleton**:
1. Participants — Who was there.
2. Key Points — Bullet summary.
3. Decisions Made — What was decided, by whom.
4. Action Items — Person + task + deadline.
5. Open Questions — Unresolved items.

### note
**Receiver**: Self (temporary)
**Goal**: Don't lose the signal
**Skeleton**:
1. Context — One line (what triggered this).
2. Content — The information.
3. Route — Where this should live permanently.

### decision-log
**Receiver**: Future self, team
**Goal**: Record decision + rationale
**Skeleton**:
1. Decision — What was decided.
2. Context — Why now, what triggered it.
3. Options Considered — What else was on the table.
4. Rationale — Why this option won.
5. Implications — What changes as a result.
6. Reversibility — How hard to undo.

### pitch
**Receiver**: Prospects, partners, investors
**Goal**: Create desire for action
**Skeleton**:
1. Hook — Problem statement that resonates.
2. Solution — How we solve it.
3. Proof — Evidence it works.
4. Ask — What we want from them.

### report
**Receiver**: Stakeholders, team
**Goal**: Inform on status/progress
**Skeleton**:
1. Summary — Key findings in 3 bullets.
2. Details — Section per topic area.
3. Metrics — Numbers with context.
4. Recommendations — What to do next.

## Receiver-Genre Matrix

| Receiver | Primary Genres | NEVER Send |
|----------|---------------|------------|
| Sales Rep | brief, pitch | spec, technical docs |
| Lead Engineer | spec + explicit constraints | open-ended requirements |
| Developers | spec, readme, changelog | business strategy |
| Course Partner | brief, proposal, pitch | raw specs |
| Content Lead | brief, pitch, social-post | technical specs |
| Sales Closers | brief, pitch | anything technical |
| Technical Partner | spec, runbook | fluff, sales content |
| The Operator | spec, plan, review, brief | — (anything) |

## Quality Checklist (Before Sending)
- [ ] Does every sentence carry actionable intent? Cut filler.
- [ ] Is the genre right for this receiver?
- [ ] Is the structure clear? Can they extract action items immediately?
- [ ] Does output density match receiver bandwidth?
- [ ] Is there unnecessary hedging? BE DIRECT.
- [ ] Is there repetition? CONSOLIDATE.
